# CPE 子网测试工具 使用说明（小白版）

这是一个**两台电脑之间自动测网**的工具：自动 Ping 测连通，可选择 iperf3 或
Microsoft ctsTraffic 灌包测速，并自动生成一份 HTML 报告。**不需要懂编程，也不需要装 Python。**

---

## 一、准备东西（一次性）

1. 两台 Windows 10 或更高版本的电脑（只用 Ping/iperf3 时也可使用项目支持的其他平台）：
   - **主控机**：你操作键盘的这台，发起测试
   - **辅测机**：另一台，跑起来就不用管
2. 两台电脑都接到被测 CPE / 路由器的子网里（网线或 WiFi 都行）
3. 把同一个 v4.2.1 Windows Release 文件夹完整复制到两台电脑：

   ```
   cpe_test.exe          本工具（就一个文件）
   iperf3.exe            灌包工具（从 iperf 官网 iperf.fr 或 GitHub 搜 iperf3-win-builds 下载）
   ctsTraffic.exe        Windows 专用灌包工具（官方 v4.2.1 Windows 包已带）
   start_agent.bat       辅测机双击这个
   start_master_select_config.bat  主控机双击这个（选择本次测试网口）
   configs\              各网口的具名 JSON 配置
   ```

   > 只测 Ping 或只测 ctsTraffic 时，iperf3.exe 可以不放。只测 Ping 时，两种灌包工具都不需要。
   > 主控机和辅测机必须使用同一个 `cpe_test` 版本；不要只替换其中一台的 exe。

---

## 二、开始测试（每次 3 步）

### 第 1 步：辅测机上双击 `start_agent.bat`

窗口会显示：

```
本机 IP 列表（把主控能 ping 通的那个告诉主控机）:
    以太网 = 192.168.1.3
    WLAN  = 10.228.46.50
agent 已启动，监听 0.0.0.0:28801
等待主控连接...
```

- **记下窗口里显示的 IP**（选主控能连上的那个，一般是 10.x 的管理口）
- 弹出防火墙提示时，**全部勾选并点"允许访问"**
- 这个窗口**不要关**，测试全程放着

### 第 2 步：主控机上双击 `start_master_select_config.bat`

- 先按菜单选择本次要测的网口（SGMII / Wi-Fi 5G / 10GUSB / 全部常用网口）
- 提示 `请输入辅测机 IP` 时，输入第 1 步记下的 IP，回车
- 程序会自动扫出两台电脑的网卡，比如：

  ```
  【主控】 DESKTOP-A：
    [1] SGMII2.5G  以太网    192.168.1.2   2500Mbps
  【辅测】 DESKTOP-B：
    [1] SGMII2.5G  以太网    192.168.1.3   2500Mbps
  ```

### 第 3 步：一路回车

- 菜单每一步都有推荐默认值（带 `*` 号的），**不知道选什么就直接回车**
- 随包具名配置使用 `pairs` 批量模式，会直接生成任务，不再询问“测试类型”；默认测试 iperf3 和 Ping
- 想不走自动化、自己指定原生 ctsTraffic 参数打流，请看下文“不走 cpe_test”章节；ctsTraffic 仅适用于 Windows 10+ 电脑
- 选完后显示任务清单，回车开始跑
- 跑完自动弹出 HTML 报告；PASS、性能不达标、仅测量、负载不足等状态会分开显示

## 三、看结果

| 文件 | 说明 |
|------|------|
| `report_日期时间.html` | 测试报告（自动打开；发给别人时连同 iperf_outputs 文件夹一起拷） |
| `master_日期时间.log` | 主控运行进度、摘要、错误和原始文件路径 |
| `task_results.json` | 结果库，开 RESUME 后 24 小时内 PASS 的任务自动跳过 |
| `iperf_outputs/iperf_raw_*.log` | iperf3 每条流的 client/server/事件及所有重试原文 |
| `iperf_outputs/ctstraffic_raw_*.log` | ctsTraffic client/server 命令、输出和解析摘要 |
| `iperf_outputs/nic_samples_*.csv` | OS 网卡累计计数器的逐样本 RX/TX 原始记录 |
| `iperf_outputs/screenshot_*.png` | 可选截图附件 |

报告里关键列：
- **结果 / 原因码 / 原因详情**：区分真正性能不达标与流没连够、有效时间不足、采样失败
- **请求/活跃/要求流、重试**：例如 2 条只通 1 条会直接显示为 2/1/2
- **目标、TX 均值/P10**：确认发送端确实给到了足够负载
- **接收网卡平均/P10/中位/P95/最低/最高 Mbps**：共同有效窗口内的接收端实测吞吐
- **有效/要求秒、采样覆盖率**：确认“测 3 分钟”是真正取得 180 秒稳态数据
- **后端发送/接收 Mbps**：用于确认工具确实起流和辅助诊断；正式目标吞吐仍看网卡数据
- **UDP 丢包率 / Ping 丢包率**：丢包情况

结果不要只看成简单的绿/红：

| 结果 | 含义 |
|------|------|
| `PASS` | 有明确目标，负载、窗口、速率和丢包条件都满足 |
| `RATE_FAIL` | CPE 接收速率/UDP 丢包确实不达标，或 UDP 单流安全耗尽全部尝试后仍未灌通 |
| `UNSTABLE` | 平均值达标，但 5 秒滚动 P10 低，存在明显掉速 |
| `MEASURED` | 理论目标未知，只报告实际能力，不伪造 PASS |
| `NOT_EVALUATED` | 流没连够、发送负载不够、有效窗口或采样覆盖不足，不能评价 CPE 性能 |
| `SETUP_ERROR` | 吞吐工具、server/client、平台、参数、状态查询或资源清理等环境搭建失败 |

## 四、网卡速率监控（不需要两台电脑，单机就能用）

`cpe_test monitor` 是一个**单独的网卡测速工具**，不需要辅测机、不需要 iperf3，
就是对着你本机的一块网口看实时流量。适合调试、观察背景流量、验证打流结果。

### 怎么用

```bash
# 第一次用：列出网卡让你选
cpe_test monitor

# 之后直接回车用上次选的网卡，想换就输序号
cpe_test monitor

# 命令行一键启动
cpe_test monitor -n "以太网" -i 1 -c speed_log.csv
```

**参数说明：**

| 参数 | 简写 | 含义 | 默认 |
|------|------|------|------|
| `--iface NAME` | `-n` | 网卡名（如"以太网"） | 上次选的或交互选 |
| `--interval N` | `-i` | 采样间隔（秒） | 1 |
| `--duration N` | `-d` | 跑多久自动停（0=不限） | 0 |
| `--csv FILE` | `-c` | 输出 CSV 文件路径 | 不输出 |

### 运行时显示

```
网卡: [以太网]  间隔: 1s  按 Ctrl+C 停止

时间          速率(Mbps)
--------------------------
12:00:01        1690.10
12:00:02        1688.50
^C
==================================================
网卡: 以太网
时长: 2s (2 次采样)
平均: 1689.30 Mbps
峰值: 1690.10 Mbps
CSV : speed_log.csv
```

### CSV 文件

按 Ctrl+C 结束后，自动在文件顶部写入平均值和峰值摘要（Excel 打开一目了然）：
```
# Average (Mbps),1689.30
# Peak (Mbps),1690.10
Time,Speed(Mbps)
12:00:01,1690.10
12:00:02,1688.50
```

### 跟子网测试的区别

| | `cpe_test monitor` | `cpe_test master` |
|---|---|---|
| 需要几台电脑 | 1 台 | 2 台 |
| 需要吞吐工具 | 不需要 | 依所选后端需要 iperf3 或 ctsTraffic；纯 Ping 不需要 |
| 做什么 | 看网口实时流量 | 自动化灌包+ping测试 |
| 输出 | CSV（可按 Ctrl+C 随时停） | HTML 报告 |

## 五、配置文件（可选，进阶）

把 `config.example.json` 改名为 `config.json` 放到 exe 同目录，就能：
- 固定辅测机 IP（不用每次输）
- 固定测试项，双击就全自动跑（配合 `--auto` 可完全免交互）

**不同网口请用不同配置文件，不要靠注释切换。** JSON 规范不支持 `//`、`#` 或
`/* ... */` 注释。本发布包的 `configs` 目录已经提供：
`config-sgmii.json`、`config-wifi5g.json`、`config-10gusb.json` 和
`config-all-common.json`。主控机双击 `start_master_select_config.bat` 即可选择；
也可执行 `start_master.bat configs\\config-sgmii.json` 指定文件。

**换一套电脑不用改配置**：`"master:SGMII2.5G"` 这种写法是按"网卡角色"找网卡的，
今天 IP 是 192.168.1.100、明天换成 192.168.5.200，程序自动适配。

常用字段（完整示例见 config.example.json）：

| 字段 | 含义 | 默认 |
|------|------|------|
| `agent_host` | 辅测机 IP | 空=启动时询问 |
| `ipv4_prefixes` | 测试网段前缀，扫网卡时只认这些 | `["192.168."]` |
| `iperf.duration` | 全局吞吐时长默认值；iperf3/CTS 的具体映射见下文，必须大于 0 | 180 |
| `iperf.tcp_windows` | TCP socket buffer 档位；CTS 会映射到发送/接收方向 | 64k / 1m / 4m |
| `iperf.udp_profiles` | UDP 档位；每项可写 `bandwidth`、`length`、`window` | 1m/100m/500m/1000m(小包)/2500m |
| `iperf.rate_check.mode` | `auto` / `verify` / `observe` / `discover` | auto |
| `iperf.rate_check.flow_retries` | UDP 额外重试预算；iperf3 单流和 CTS `Connections:1` 每方向总尝试数为 `max(flow_retries + 1, 3)` | 1 |
| `iperf.rate_check.min_active_ratio` | 正式计分所需活跃流比例；另有最低 2 流保护 | 0.9 |
| `ping.count` | ping 包数 | 100 |
| `ping.payload_sizes` | 子网 Ping 负载字节 | [32, 1600, 65500] |
| `ctstraffic.udp_frame_rate` | CTS UDP MediaStream 每秒帧数 | 100 |
| `ctstraffic.udp_buffer_depth_secs` | CTS UDP 应用层缓冲深度（秒，不是 socket buffer） | 1 |
| `ctstraffic.status_update_ms` | CTS 状态输出周期（毫秒） | 1000 |
| `tests[].src / dst` | `master:角色` 或 `master:NAME=接口名` | — |
| `tests[].direction` | `A->B` / `B->A` / `bidir`(双向并发) | A->B |
| `tests[].kinds` | `iperf`、`ctstraffic`、`ping` 可任选或组合 | ["iperf"] |
| `tests[].streams` | 并发流数；CTS 映射成 `Connections`，一个进程承载多连接 | 1 |
| `tests[].iperf_duration` | 历史字段名，当前供 iperf3 与 CTS 共用，不能为 0 | 全局值 |
| `tests[].rate_mode` | 单场景覆盖速率模式 | 全局值 |
| `tests[].rate_targets_mbps` | 已知目标；单向写 forward，双向写 ab/ba | 未知 |
| `resume` | 24h 内 PASS 过的任务跳过 | false |
| `screenshot` | 每个灌包任务后接收端截图 | false |

### pairs 自动配对（最省事的写法）

如果测试模式是"主控上所有网口 × 辅测上所有网口，两两互测"，不用一个个写 tests[]，
用 `pairs` 写一行就行：

```json
{
  "pairs": "all",
  "universal_params": {
    "directions": ["A->B", "bidir"],
    "kinds": ["iperf", "ctstraffic", "ping"],
    "transports": ["tcp", "udp"],
    "ip": ["v4"],
    "streams": 1,
    "iperf_duration": 180,
    "rate_mode": "auto"
  }
}
```

- `"pairs": "all"` — 程序自动把所有网口配对，同 UNKNOWN 角色跳过
- `"pairs": [{"master":"SGMII2.5G","agent":"SGMII2.5G"}]` — 也可以手动指定角色对
- `universal_params` — 统一测试参数，字段和 tests[] 一样，不写走全局默认
- 注意：`tests[]` 有内容时不走 pairs，二者选一

角色说明：`10GUSB`（高速 USB/NCM，兼容 4.2G 显示 bug）/ `10GETH` / `SGMII2.5G` /
`SGMII1G` / `RNDIS` / `WIFI5G` / `WIFI2.4G`。RNDIS 关键字优先，不能因为描述里同时出现 USB/10G 就误当成高速 NCM。

### ctsTraffic 要怎么用

**如果你说的“手动”是“不走自动化，直接指定原生 CTS 参数打流”，当前 `cpe_test`
做不到。** 它没有 `cpe_test ctstraffic ...` 直通子命令；交互菜单仍会自动生成
`Listen`、`Target`、`Bind`、端口和 server/client 作业。要完全手工控制参数，请直接
运行随 Windows Release 提供的 `ctsTraffic.exe`。下面先给出可以直接照抄修改的原生命令，
后面再说明 `cpe_test` 的交互式自动编排入口。

#### 不走 cpe_test：直接用原版 ctsTraffic 手工打流

本节按项目固定捆绑的 **Microsoft ctsTraffic 2.0.4.0 x64** 编写。先约定一个双机例子：

| 电脑 | 测试网卡 IP | 记号 |
|------|-------------|------|
| A | `192.168.1.10` | A |
| B | `192.168.1.20` | B |

请把示例 IP 换成两台电脑真实的测试网卡 IP。两边都要把同一份 `ctsTraffic.exe` 放到
当前目录，允许 Windows 防火墙放行该程序和所用端口。**先启动所有 `-Listen` 命令，
再启动 `-Target` 命令。** 原生命令的帮助可直接查看：

```bat
ctsTraffic.exe -Help
ctsTraffic.exe -Help:Tcp
ctsTraffic.exe -Help:Udp
ctsTraffic.exe -Help:Logging
ctsTraffic.exe -Help:Advanced
```

##### TCP 单向 A → B：1 条连接，运行约 30 秒

TCP `Pattern:Push` 的数据方向是 **client → server**。先在 B 上启动接收端：

```bat
ctsTraffic.exe -Listen:192.168.1.20 -Port:56000 -Protocol:TCP -Pattern:Push -Verify:connection -Buffer:1048576 -Transfer:0xffffffffffffffff -RecvBufValue:1048576 -ConsoleVerbosity:1 -StatusUpdate:1000
```

再在 A 上启动发送端：

```bat
ctsTraffic.exe -Target:192.168.1.20 -Bind:192.168.1.10 -Port:56000 -Protocol:TCP -Pattern:Push -Verify:connection -Buffer:1048576 -Transfer:0xffffffffffffffff -SendBufValue:1048576 -Connections:1 -Iterations:1 -TimeLimit:30000 -ConsoleVerbosity:1 -StatusUpdate:1000
```

- `Transfer:0xffffffffffffffff` 让这条连接持续传输；`TimeLimit:30000` 是 client 的
  30000 毫秒硬截止。原版帮助明确说明：命中 `TimeLimit` 可能记录 time-limit 错误，
  这是该“固定时间测速”写法的预期收尾，不要把这一行本身误判成网络故障。
- `Connections:1 -Iterations:1` 表示这次只建立 1 条连接；若它提前失败，原版不会在
  同一轮自动补建新连接。修复环境后请由人工重新运行整组命令。
- client 停止后，B 上的监听进程可能继续等待下一条连接；查看完最终统计后在 B 窗口按
  `Ctrl+C` 安全停止。想完全手工控制时长，也可以从 client 命令删除 `TimeLimit`，
  需要结束时先在 client 按 `Ctrl+C`，再停止 server。
- 想限制 TCP 发送速率，可只在实际发送端增加 `-RateLimit:62500000`。原版 CTS 的
  `RateLimit` 单位是**每连接 bytes/s**，所以 `62500000 × 8 = 500000000 bit/s`，
  即每连接约 500 Mbps；不写就是不限速。
- 想按固定数据量正常完成而不是命中时间上限，可把两端 `Transfer` 都改为例如
  `4294967296`（每连接 4 GiB），client 改为 `Iterations:1` 并删除 `TimeLimit`，
  server 增加 `ServerExitLimit:1`。

要测 B → A，有两种方法：交换 client/server 角色和 IP；或者保持 A 为 client、B 为
server，并把**两端**的 `Pattern:Push` 都改成 `Pattern:Pull`。`Pull` 表示 server 发、
client 收，此时可选的 `SendBufValue` 应放在 B，`RecvBufValue` 应放在 A。

##### TCP 双向：1 条全双工连接

原版 CTS 自带 `Pattern:Duplex`，一条 TCP 连接里双方同时发送和接收。先在 B 运行：

```bat
ctsTraffic.exe -Listen:192.168.1.20 -Port:56002 -Protocol:TCP -Pattern:Duplex -Verify:connection -Buffer:1048576 -Transfer:0xfffffffffffffffe -SendBufValue:1048576 -RecvBufValue:1048576 -ConsoleVerbosity:1 -StatusUpdate:1000
```

再在 A 运行：

```bat
ctsTraffic.exe -Target:192.168.1.20 -Bind:192.168.1.10 -Port:56002 -Protocol:TCP -Pattern:Duplex -Verify:connection -Buffer:1048576 -Transfer:0xfffffffffffffffe -SendBufValue:1048576 -RecvBufValue:1048576 -Connections:1 -Iterations:1 -TimeLimit:30000 -ConsoleVerbosity:1 -StatusUpdate:1000
```

Duplex 的 `Transfer` 是双向合计字节数，内部会平均分给发送和接收；它必须使用偶数。
这里故意写最大偶数 `0xfffffffffffffffe`，不要照搬单向 Push 的奇数最大值
`0xffffffffffffffff`，否则固定版本内部把奇数加 1 时会回绕成 0，任务可能立即结束。

这是一条原生全双工连接。如果你要严格模拟 `cpe_test` 报告中的“AB、BA 各一条独立
Push 流”，不要用 `Duplex`：同时运行两组上面的 TCP `Push` 命令即可，A → B 使用
端口 `56000`，再把 A/B 角色完全交换并让 B → A 使用另一个端口 `56001`。两个方向
必须使用不同端口，四个 CMD 窗口都要保持运行。A → B 继续使用前文两条命令；B → A
额外在 A 上先启动接收 server：

```bat
ctsTraffic.exe -Listen:192.168.1.10 -Port:56001 -Protocol:TCP -Pattern:Push -Verify:connection -Buffer:1048576 -Transfer:0xffffffffffffffff -RecvBufValue:1048576 -ConsoleVerbosity:1 -StatusUpdate:1000
```

再在 B 上启动发送 client：

```bat
ctsTraffic.exe -Target:192.168.1.10 -Bind:192.168.1.20 -Port:56001 -Protocol:TCP -Pattern:Push -Verify:connection -Buffer:1048576 -Transfer:0xffffffffffffffff -SendBufValue:1048576 -Connections:1 -Iterations:1 -TimeLimit:30000 -ConsoleVerbosity:1 -StatusUpdate:1000
```

严格双向的启动顺序是：B 的 `Listen:56000`、A 的 `Listen:56001` 都准备好以后，再启动
A 的 `Target:56000` 和 B 的 `Target:56001`。

##### UDP 单向 A → B：1 条流，500 Mbps，1400 字节 datagram，30 秒

CTS UDP 的角色与 TCP Push 相反：**`-Listen` 的 server 才是数据发送端，
`-Target` 的 client 是接收端。** 因此 A → B 时，先在 A 上启动发送 server：

```bat
ctsTraffic.exe -Listen:192.168.1.10 -Port:56100 -Protocol:UDP -Verify:connection -BitsPerSecond:500000000 -FrameRate:100 -StreamLength:30 -DatagramByteSize:1400 -SendBufValue:1048576 -ServerExitLimit:1 -ConsoleVerbosity:1 -StatusUpdate:1000
```

再在 B 上启动接收 client：

```bat
ctsTraffic.exe -Target:192.168.1.10 -Bind:192.168.1.20 -Port:56100 -Protocol:UDP -Verify:connection -BitsPerSecond:500000000 -FrameRate:100 -BufferDepth:1 -StreamLength:30 -DatagramByteSize:1400 -RecvBufValue:1048576 -Connections:1 -Iterations:1 -ConsoleVerbosity:1 -StatusUpdate:1000 -JitterFilename:udp_ab_jitter.csv
```

这里有四个关键点：

- 原版 `BitsPerSecond` 只接受整数 bit/s：500 Mbps 要写 `500000000`，不能写
  `500m`。`500m` 是 `cpe_test` 配置层提供的友好格式，程序会在内部转换。
- `BitsPerSecond` 是**每条 UDP connection/stream** 的速率。若 client 改为
  `Connections:5`，总提供速率约为 `500 Mbps × 5 = 2500 Mbps`；server 的
  `ServerExitLimit` 也应相应改为 5。
- `FrameRate:100` 是每秒媒体 frame 数，不等于每秒只有 100 个 UDP 包。CTS 会先算
  `frame size ≈ BitsPerSecond ÷ 8 ÷ FrameRate`，再按 `DatagramByteSize` 把一个
  frame 拆成多个 datagram。上例每个 frame 约 625000 字节，会拆成很多 1400 字节包。
- `BitsPerSecond`、`FrameRate`、`StreamLength`、`Protocol`、`Port` 和 `Verify` 必须
  在两端一致；若 server 指定 `DatagramByteSize`，client 的值必须不小于 server，
  最简单就是两端写相同值。
- A → B 指主要负载数据方向；UDP client 仍会向 server 发送 START、会话建立等少量控制报文，
  所以抓包时看到少量 B → A 报文是正常的。

##### UDP 双向：AB、BA 每方向各 1 条流

UDP 没有 `Pattern:Duplex`。双向必须同时运行两组 MediaStream，并使用不同端口。
先保留上面的 A → B 两条命令（端口 `56100`），另外打开两个 CMD 窗口：

先在 B 启动 B → A 的发送 server：

```bat
ctsTraffic.exe -Listen:192.168.1.20 -Port:56101 -Protocol:UDP -Verify:connection -BitsPerSecond:500000000 -FrameRate:100 -StreamLength:30 -DatagramByteSize:1400 -SendBufValue:1048576 -ServerExitLimit:1 -ConsoleVerbosity:1 -StatusUpdate:1000
```

再在 A 启动 B → A 的接收 client：

```bat
ctsTraffic.exe -Target:192.168.1.20 -Bind:192.168.1.10 -Port:56101 -Protocol:UDP -Verify:connection -BitsPerSecond:500000000 -FrameRate:100 -BufferDepth:1 -StreamLength:30 -DatagramByteSize:1400 -RecvBufValue:1048576 -Connections:1 -Iterations:1 -ConsoleVerbosity:1 -StatusUpdate:1000 -JitterFilename:udp_ba_jitter.csv
```

正确启动顺序是：A 的 `Listen:56100`、B 的 `Listen:56101` 都准备好以后，再分别启动
B 的 `Target:56100` 和 A 的 `Target:56101`。这样就是 AB、BA 各一条独立 UDP 流。

##### 原版 ctsTraffic 常用参数解释

通用参数：

| 原生参数 | 作用 | 容易写错的地方 |
|----------|------|----------------|
| `-Listen:<IP或*>` | 作为 server 在指定地址监听；`*` 表示所有地址 | 与 `Target` 互斥；多网卡测试建议写明确 IP |
| `-Target:<IP或主机名>` | 作为 client 连接 server | 可以重复指定多个 target；UDP client 仍是接收数据的一方 |
| `-Bind:<本地IP>` | client 发起连接前绑定本地网卡 | 多网卡机器强烈建议写，避免流量走错接口；不能和 `Listen` 同时用 |
| `-Port:N` | 测试数据端口 | 两端必须一致；默认 4444，示例改用 56000/56100 段；双向独立流必须分端口 |
| `-Protocol:TCP/UDP` | 选择协议 | 两端必须一致；默认 TCP |
| `-Verify:data/connection` | `data` 校验每个接收缓冲区；`connection` 只校验连接/字节或帧语义，CPU 开销较低 | 两端必须一致；默认 `data`，高吞吐摸底可先用 `connection` |
| `-ConsoleVerbosity:0..5` | 控制控制台输出；1=状态，5=连接+错误+状态 | 默认 4；测速时通常用 1 |
| `-StatusUpdate:N` | 状态刷新周期，单位毫秒 | 原版只要求大于 0，默认 5000；示例用 1000 |
| `-ConnectionFilename:F` | 逐连接日志 | `.csv` / 普通文本按官方日志规则输出 |
| `-StatusFilename:F` | 周期状态日志 | 双向多个进程不要共用同一个文件名 |
| `-ErrorFilename:F` | 错误日志 | 建议长时间测试时开启 |
| `-SendBufValue:N` / `-RecvBufValue:N` | 设置 Winsock `SO_SNDBUF` / `SO_RCVBUF`，单位字节 | 不是 TCP 拥塞窗口；原版写整数，`1048576` 才是 1 MiB，不能写 `1m` |

TCP 参数：

| 原生参数 | 作用 | 单位/默认 |
|----------|------|-----------|
| `-Pattern:Push` | client 发、server 收 | TCP 默认 |
| `-Pattern:Pull` | server 发、client 收 | 两端都必须写 Pull |
| `-Pattern:Duplex` | client/server 同一连接双向同时发收 | 两端都必须写 Duplex |
| `-Connections:N` | client 同时维持的连接数 | client 专用；默认 8；原版不受 cpe_test 的 32 流上限约束，但会受机器资源限制 |
| `-Iterations:N` | 对 Connections 这一批连接迭代几次 | client 专用；总连接数=`Connections × Iterations`；0=无限 |
| `-Transfer:N` | 每条 TCP 连接传输的总字节数 | 两端必须一致；默认 1073741824（1 GiB）；Push/Pull 可用 `0xffffffffffffffff`，Duplex 必须写偶数，持续流用 `0xfffffffffffffffe` |
| `-Buffer:N` | 每次 send/recv 的应用层 I/O buffer | 默认 65536 字节；与 `SendBufValue/RecvBufValue` 不是一回事 |
| `-RateLimit:N` | 每条连接的 TCP 限速 | **bytes/s**，不是 bit/s；不写=不限速 |
| `-TimeLimit:N` | 进程最大运行时间 | 毫秒；0 非法；命中属于硬中止并可能记录错误，适合测速硬截止而不是“正常完成”条件 |
| `-ServerExitLimit:N` | server 处理 N 条连接后退出 | server 专用；0/省略=无限等待 |

UDP 参数：

| 原生参数 | 作用 | 单位/默认 |
|----------|------|-----------|
| `-BitsPerSecond:N` | 每条 UDP stream 的发送速率 | 必填、整数 bit/s；两端一致；不能写 k/m/g 后缀，非 8 的倍数会向下对齐到整字节 |
| `-FrameRate:N` | 每秒媒体 frame 数 | 必填、两端一致；不是 packet rate |
| `-StreamLength:N` | 每条 stream 持续时间 | 必填、秒、两端一致 |
| `-BufferDepth:N` | client 预缓冲和丢帧判定观察深度 | 秒；默认 1；主要影响接收 client，不是 socket buffer |
| `-DatagramByteSize:N` | 每个 UDP datagram 的最大字节数 | 固定版本还要容纳协议头、连接 ID 和分片余数修正，通用安全值应为 `52..=65507`；通常保持 1400，避免超过 MTU；client 值必须 ≥ server 值 |
| `-Connections:N` | client 同时建立的 stream 数 | 每条都按完整 BitsPerSecond 发送；UDP 默认 1 |
| `-Iterations:N` | 这一批 streams 运行几轮 | client 专用；一次有界测试通常写 1 |
| `-ServerExitLimit:N` | server 服务 N 条 stream 后退出 | 单流一次写 1；5 条并发一次写 5 |
| `-JitterFilename:F` | client 的逐 datagram 时间戳/jitter 日志 | 仅 UDP client 可用 |

原版数值参数大多要求写**整数原始单位**，不像 `cpe_test` 配置会接受 `500m`、`1m`
等友好后缀。如果 UDP 参数报错，还要检查
`BitsPerSecond ÷ 8 ÷ FrameRate` 算出的 frame 是否至少 40 字节，以及
`BufferDepth × FrameRate`、`StreamLength × FrameRate` 是否溢出。

##### 怎么判断原生命令是否真的灌通

- 先看两端开头的 `Configured Settings`，确认 IP、端口、协议、方向、连接数和速率没有写错。
- TCP Push 时，client 的 `SendBps` 和 server 的 `RecvBps` 是 **bytes/s**；换成 Mbps 用
  `Bps × 8 ÷ 1000000`。`In-Flight` 应出现预期连接数。
- UDP 重点看接收 client 的 `Bits/Sec`、`Streams`、`Completed`、`Dropped`、
  `Repeated`、`Errors`；最终 `Total Successful Frames` 应大于 0。
- UDP 和“有限 Transfer、自然完成”的 TCP 测试中，最终 `SuccessfulConnections` 应
  达到预期，`NetworkErrors` 和 `ProtocolErrors` 应为 0；UDP 还应核对
  dropped/duplicate/error frames。
- 使用巨大 `Transfer + TimeLimit` 的 TCP 硬截止摸底时，远端可能把未完成 Transfer 的
  EOF 记为 `TooFewBytes` / `ProtocolErrors`。这类测试应看截止前状态行是否持续
  `In-Flight=1`、Send/RecvBps 大于 0 且窗口内无 Net/DataError；想要干净的成功摘要，
  请改用前文“有限 Transfer + Iterations:1 + ServerExitLimit:1”的自然完成方式。
- 原版手工模式不会替你做 `cpe_test` 的三次安全重试、NIC 采样、HTML 报告和自动 PASS/FAIL。
  单流没有任何 CTS 自身速率/字节/frame 证据时，请完整停止两端、确认进程退出后重新运行；
  连续 3 次仍无测量即可按灌流失败处理。

[官方 streaming 示例](https://github.com/microsoft/ctsTraffic/blob/4468b8085b6ac58c531596216f10e812287174ea/TestScripts/streaming.txt)
以及项目固定 2.0.4.0 提交中的 `-Help` 定义整理。

#### 如果改回由 cpe_test 自动编排

需要自动重试、双端进程清理、NIC 采样、HTML 报告和统一 PASS/FAIL 时，再使用
`cpe_test.exe agent` 与 `cpe_test.exe master --agent-host <辅测机IP>`，在交互菜单选择
ctsTraffic。这个入口仍是自动编排，不是原生命令透传；要看到完整测试类型菜单，加载的
配置不能含 `pairs`，且主控不能加 `--auto`。随包 `start_master_select_config.bat` 使用
`pairs` 批量配置，当前默认只生成 iperf3 和 Ping；批量启用 CTS 的写法见下文。

#### 在批量配置里启用 CTS

发布包里的具名配置为了保持升级前的默认测试量，仍然使用 `"kinds": ["iperf", "ping"]`。
只跑 CTS 时改为：

```json
"kinds": ["ctstraffic"]
```

想把两种后端放在同一份报告中对比时改为：

```json
"kinds": ["iperf", "ctstraffic", "ping"]
```

使用 CTS 前请记住下面几件事：

- **系统**：主控和辅测都必须是 Windows 10 或更高版本；程序会检查两端真实 Windows major 版本，版本无法确认时 fail-closed。两边都要有 `ctsTraffic.exe`，并使用同一个 `cpe_test` Release。
- **不会偷偷换工具**：平台、版本能力或二进制不满足时，CTS 单元会明确显示 `SETUP_ERROR / CTSTRAFFIC_PREFLIGHT_FAILED`，不会自动改跑 iperf3；同一配置里的 iperf3 和 Ping 仍会继续。
- **方向**：报告里的 `src → dst` 永远是数据方向。CTS TCP 是 `src` 上的 client 发、`dst` 上的 server 收；CTS UDP 恰好相反，由 `src` 上的 server 发、`dst` 上的 client 收。程序会自动安排，用户不用把 src/dst 倒过来。
- **流数**：TCP 的 `streams: 5` 会直接变成 `Connections:5`；UDP 开启链路裁剪时，先按每流速率和每个方向的路径上限计算实际连接数，再映射为 `Connections`。一个 CTS 进程承载多条连接。CTS 自动化只接受 `1..=32`；配置文件越界会记录 `SETUP_ERROR / CTSTRAFFIC_ARGS_INVALID`，交互式输入会要求重输。
- **时长**：配置仍使用历史字段 `iperf_duration`。TCP 映射到 `TimeLimit` 硬截止，UDP 映射到 `StreamLength`。CTS 自动化只接受 `1..=86400` 秒；配置文件中的 `0` 会记录 `SETUP_ERROR / CTSTRAFFIC_ARGS_INVALID`，交互式输入会要求重输。CTS 原生无限运行只能手工执行并手工停止。
- **buffer**：TCP 使用 `tcp_windows`，UDP 使用 profile 里的可选 `window`。CTS 会把它们设到实际发送/接收方向的 `SendBufValue`/`RecvBufValue`；iperf3 UDP 则生成 `-w`。这里说的是 socket buffer，不是 TCP 拥塞窗口。
- **UDP 速率**：`bandwidth` 是每条连接的速率；未裁剪时 `500m × 5 streams` 的总提供速率约为 2500 Mbps，发生裁剪时必须改用“500m × 实际 Connections”计算。CTS 要求完整格式为十进制数值加可选 `k/m/g` 或 `kbps/mbps/gbps` 后缀，尾随字符、科学计数或混合后缀会记录 `SETUP_ERROR / CTSTRAFFIC_ARGS_INVALID`。默认 CTS UDP `FrameRate=100`、`BufferDepth=1`，属于 MediaStream 模型，不等同于 iperf3 的 UDP flood。
- **单连接硬门槛**：CTS UDP `Connections:1` 与 iperf3 UDP 单流一样，每个方向总尝试数为 `max(flow_retries + 1, 3)`；双向 AB、BA 各自独立并行。每轮都必须完整启动 server/client，并确认两端停止和清理成功后才能复用端口。
- **灌通证据**：只能由 CTS 自身的 rate、bytes、successful frames 等输出证明连接已建立；NIC 上即使有背景流量也不能冒充 CTS 灌通。NIC 只在工具已证明起流后用于验证正式目标吞吐和采样覆盖。
- **失败边界**：全部尝试均安全完成仍无 CTS 自身测量时为 `RATE_FAIL / CTSTRAFFIC_SINGLE_UDP_STREAM_FAILED`。平台不支持、缺工具、非法参数、显式取消、进程启动/状态查询或清理未确认等搭建问题仍是 `SETUP_ERROR`；一旦已有工具测量，server 在显式停止前异常退出/超时或其他 CTS 运行错误会记录 `RATE_FAIL / CTSTRAFFIC_RUNTIME_ERRORS`，并按真实 UDP 丢帧和目标速率判定，不靠继续重试掩盖结果。
- **不同模型**：CTS 不套用 iperf3 的逐进程多流调度、settle、5 秒滚动 P10 与分阶段 `discover`；CTS 的 `discover` 当前按固定 `Connections` 的能力测量记录为 `MEASURED`。两者只共享上述单流硬门槛和安全生命周期原则。

带 UDP socket buffer 的 profile 示例：

```json
{"bandwidth": "500m", "length": "1400", "window": "1m"}
```

`window` 可写 `64k`、`1m`、`1.5m` 等；不写就保持旧配置行为。iperf3 与 ctsTraffic
的连接模型、截止方式和 UDP 帧模型不同，两组数值适合并列观察，不应当作完全相同语义的互换结果。

### UDP 单流为什么至少尝试 3 次

iperf3 UDP 单向只有 1 流、CTS UDP 单向只有 `Connections:1`，以及双向测试的 AB/BA
每方向各 1 流时，都属于硬连通门槛。每个方向的总尝试数是
`max(flow_retries + 1, 3)`；双向两个方向各自拥有完整预算并同时运行，而不是共用三次。

每轮都必须完整启动 server/client，并在停止、进程回收和输出线程清理全部确认后才进入
下一轮。判断“灌通”只认所选工具自身的 rate、bytes、frame/datagram 等输出，NIC 背景流量
不能补成成功。全部安全尝试仍没有工具测量时，iperf3 为
`RATE_FAIL / SINGLE_UDP_STREAM_FAILED`，CTS 为
`RATE_FAIL / CTSTRAFFIC_SINGLE_UDP_STREAM_FAILED`，不会降级为 `NOT_EVALUATED`。

缺工具、平台不支持、非法参数、server/client 启动或状态查询失败、显式取消、清理未确认等
仍是 `SETUP_ERROR`。一旦某轮已有工具测量，就直接按该轮真实运行错误、UDP 丢包和目标
速率判定，不会为了得到更好结果继续重试。

### 不知道 CPE 理论速率时怎么选

保持默认 `rate_mode: "auto"` 即可：

- 明确的 EVB `10GUSB/NCM → 10GETH` 自动按 6400 Mbps 验证，`10GETH → 10GUSB/NCM` 自动按 8400 Mbps 验证。
- SGMII2.5G、SGMII1G、RNDIS、WiFi 或经过这些接口的 CPE 子网，默认进入 `observe`，报告显示 `MEASURED` 和真实速率，不会拿协商值冒充 PASS 标准。
- RNDIS 显示约 3.7G 时，发包负载最多按约 2.5G 规划；SGMII1G 按 1G；WiFi 按“协商值和 2.5G 中较小值”。
- NCM 显示 4.2G 是已知 bug，若另一端也是 EVB 10G 以太则按 10GUSB 处理；若中间经过 2.5G 以太或 WiFi，整条路径仍按较低瓶颈限流。
- 第一次想摸底“多少条流时能跑多快”，对 **iperf3 UDP** 把场景设为 `discover`；程序会逐级加流并输出活跃流数与平均/P10 RX 表。CTS 当前不会分阶段加流，使用 `discover` 时只记录本次固定 `Connections` 的测量结果。
- 产品已有正式验收线时才使用 `verify`，并写 `rate_targets_mbps`。

EVB 的两个自动目标可在 `iperf.rate_check` 中修改：

```json
"evb_usb_to_eth_target_mbps": 6400,
"evb_eth_to_usb_target_mbps": 8400
```

单个双向场景也能覆盖，例如 `src=10GUSB`、`dst=10GETH` 时写：

```json
"rate_mode": "verify",
"rate_targets_mbps": {"ab": 6400, "ba": 8400}
```

这里 `ab` 是 `src → dst`，`ba` 是 `dst → src`；如果交换了 src/dst，数值也要随方向交换。场景级目标优先于全局目标和 EVB 自动默认值。

### iperf3 的 180 秒为什么实际会跑得更久

180 秒指**双方都达到最低活跃流数后的共同有效稳态数据**。程序还需要 3 秒背景采样、流错峰启动、5 秒稳定等待、失败流重试和收尾缓冲，所以墙钟时间通常约 200 秒。报告中的“有效/要求秒”必须达到 180/180 才能正式判定。

ctsTraffic 的同一配置值语义不同：TCP 使用 `TimeLimit=180000` 毫秒硬截止，UDP 使用
`StreamLength=180` 秒；再加上 server 准备、状态轮询和收尾，墙钟时间会略长于配置值。
如果 UDP 单连接需要多轮完整尝试，墙钟时间还会按实际尝试数增加。

### 双向 5 流/2 流中，2 流方向坏 1 条怎么办

失败流会重试。若重试后两条都成功，正常进入共同窗口；若最终仍只有 1/2：

- 不会一直死等；
- 不会拿这一条流的速率判 CPE 性能失败；
- 报告为 `NOT_EVALUATED / ACTIVE_STREAMS_LOW`；
- 日志和报告会列出失败流、重试次数及错误原因。

5 条流方向默认允许 1 条最终失败（至少 4/5），20 条默认至少 18/20；但已知目标需要更多流才能提供足够负载时，以更高要求为准。

### 20/32 流为什么现在能同时跑

agent 的 16 个 HTTP worker 只负责“创建、查询、停止”后台 client job，请求会立即返回，不再被一条 180 秒 iperf3 占住。因此 20/32 条流不会在第 16 条处排队。TCP、UDP 运行日志都会按秒打印各方向 `active/connected/ended/nic-rx/iperf/err`，报告也显示请求/活跃/要求流数。

日志示例：

```text
[灌包进度][TCP][单向] active=1/1 connected=1 ended=0 nic-rx=2368.4Mbps iperf=2379.0Mbps err=0
[灌包进度][UDP][ab] active=20/20 connected=20 ended=0 nic-rx=8420.3Mbps iperf=10000.0Mbps err=0
```

这里 `nic-rx` 是操作系统网卡计数器实测值；`iperf` 只是诊断值。旧 iperf3 输出缓冲时 `iperf` 可能暂时为 `-`，不影响 `nic-rx` 正常更新。

### 报告里的 RX/P10 到底看什么

正式目标吞吐判定看的是**接收端网卡实际累计字节**，不是 iperf3/ctsTraffic 自己报的速率；但单流是否真正建立必须有工具自身证据：

- Windows 读取 `GetIfTable2`，macOS 读取 `netstat -ibn`，Linux 读取 sysfs 网卡计数器。
- iperf3 输出中的 `0.00-180.00 sec` 只用于确认真实灌包时间。旧版 iperf3 可能把所有日志在结束时一次刷出，日志到达时间只有几毫秒；程序会按 `结束秒 - 开始秒` 校准网卡采样窗口。
- ctsTraffic 的状态/summary 会解析连接数、发送/接收速率、bytes、frame 和 TCP/UDP 错误；这些工具输出用于证明 CTS 确实起流，NIC 只验证已建立流的正式目标，背景 NIC 流量不能替代工具证据。
- 网卡平均值按样本实际覆盖时间加权；P10 只使用完整覆盖的 5 秒滚动窗口。目标测试的 RX/TX 总采样和滚动窗口覆盖率都要达到 95%，否则只报 `NOT_EVALUATED`，不会误判产品性能。
- 报告里的“iperf 发送/接收 Mbps”仍会保留，方便对比 PC 或协议栈是否来不及处理，但不代替网卡 RX/P10。
- HTML 底部仍可展开原始输出；`iperf_outputs/iperf_raw_*.log` 和 `ctstraffic_raw_*.log` 分别保存两种后端原文，`nic_samples_*.csv` 保存每个网卡样本。目录名 `iperf_outputs` 为兼容旧版本保留。

### 失败重试会不会留下端口

两种 UDP 后端每轮失败后都会先取消并回收 client，再按该轮唯一 `request_id` 停止 server。只有确认 server 进程已退出、端口对应的 Child 已 `wait`、输出线程已结束，才会在同一端口开始下一次重试；停止未确认就直接禁止重试并报告 `SETUP_ERROR`。单元结束、报错或 panic 时还会按 `owner_id` 批量清理，断联时由动态 lease 最终回收。

主控与 agent 必须配套更新。开始前，主控会分别检查所选后端：iperf3 需要 agent 的
`reliable_lifecycle_v1` 能力和两端 iperf3；ctsTraffic 会要求主控真实 Windows major 版本不低于 10，且 agent 只有在自身同样通过真实版本检查时才声明
`ctstraffic_v1` 能力，然后检查两端 ctsTraffic。版本无法确认时 fail-closed。前置检查失败不会启动对应工具，也不会静默回退；
另一个吞吐后端和 Ping 不受影响，纯 Ping 仍兼容旧 agent。

### 灌包全部失败后会自动检查什么

Ping 在这里用于**子网通信测试**。常规配置默认分别发送 32、1600、65500 字节负载；只要至少收到一个来自目标的 Echo Reply 就表示子网可通信，报告仍会保留完整丢包率。

如果本轮所有 iperf3/ctsTraffic 单元都没有任何有效速率测量，例如两端缺少所选工具、旧 agent 被安全拦截、全部 server 创建失败或全部 client 未起流，主控会自动追加诊断：

- 对每个唯一 IPv4 测试方向固定发送 32 字节短 Ping，每个方向 3 包；自动诊断不发送 1600/65500。如果同方向 32 字节常规 Ping 已经执行，则不重复。
- 对涉及的每块网卡绑定该网卡源 IP，Ping 该接口自己的 IPv4 网关，判断网卡/载体是否基本正常。
- 扫描列表里的 `gw:` 是程序识别到的网关；没有网关时报告 `NOT_EVALUATED / GATEWAY_NOT_FOUND`，不会说成 100% 丢包。
- Ping 命令启动、超时、远端 HTTP/JSON 或源地址绑定错误归为 `SETUP_ERROR`；只有命令正常完成但 0 个 Echo Reply 才是 `PING_UNREACHABLE`。

当前流程不抓 PCAP。抓包更适合后续 RF 自动化在确认异常后按需开启，避免正常吞吐测试增加采集负担和排障复杂度。

命令行（给会用命令行的人）：

```
cpe_test agent                        辅测机
cpe_test master --agent-host 10.228.46.50 --auto --resume
cpe_test scan                         看本机网卡识别结果
cpe_test monitor -n "以太网" -i 1 -c r.csv   独立网卡速率监控
```

## 六、常见问题

**Q: 主控连不上辅测机？**
1. 辅测机的 agent 窗口开着吗？
2. IP 输对了吗？（用辅测机窗口里显示的）
3. 两边防火墙放行了吗？最快验证：主控 cmd 里 `ping 辅测机IP` 通不通
4. 还不行就把两台电脑的"专用/公用网络防火墙"暂时关掉试试

**Q: 提示"未找到 iperf3"？**
把 iperf3.exe（和它同包的 cygwin1.dll 等文件）放到 cpe_test.exe 同目录。

**Q: 提示 ctsTraffic 不可用或 `CTSTRAFFIC_PREFLIGHT_FAILED`？**
- 确认两台电脑都是 Windows 10 或更高版本；macOS/Linux 不支持 CTS，也不会自动回退到 iperf3。
- 确认两边 `cpe_test.exe` 来自同一个 Release，且同目录都有 `ctsTraffic.exe`。
- 官方 v4.2.1 Windows ZIP 已包含 CTS；若是自己编译，请按 `dist/ctsTraffic-请放到这里.txt` 的说明准备。

**Q: 网卡列表里少了网卡 / 是空的？**
- 网线插好了吗？WiFi 连上了吗？（断开的网卡不显示）
- 你的测试网段不是 192.168.x 开头？改 config.json 的 `ipv4_prefixes`，
  或运行 `cpe_test scan --prefix 10.10.` 验证
- 运行 `cpe_test scan` 可以随时查看识别结果

**Q: 灌包全 FAIL 但 ping 是通的？**
- 两边都放了所选后端吗（iperf3 或 ctsTraffic）？
- 防火墙拦了 56000 起的测试端口？第一次运行吞吐工具时弹出的防火墙提示要点允许
- 两端 IP 不同网段时默认不跑跨机灌包（报告里会提示），
  可在 config.json 里把 `require_same_subnet_for_iperf` 改为 false；这是兼容旧配置的字段名，当前同时约束 iperf3 和 CTS

**Q: UDP 大档位（如 2500m）没生成任务？**
默认按整条路径的可信负载上限裁剪（千兆口、RNDIS、WiFi 或接收端瓶颈都会生效）。想强行测：
交互菜单里选"UDP 限速: 关闭"，或 config.json 里 `limit_udp_by_link_speed: false`。

**Q: UDP 报 NOT_EVALUATED，是不是 CPE 不达标？**
不是。先看原因码：`ACTIVE_STREAMS_LOW` 是流没连够，`OFFERED_LOAD_LOW` 是发送负载不足，
`EFFECTIVE_WINDOW_SHORT` 是共同有效时间不足，`SAMPLE_COVERAGE_LOW` 是网卡总采样不足，
`RATE_WINDOW_COVERAGE_LOW` 是完整 5 秒滚动窗口不足。
多流场景这些条件不足时不会冒充产品性能失败。单流是明确例外：iperf3/CTS 每方向安全完成
至少 3 次仍没有工具自身测量时，会分别返回 `SINGLE_UDP_STREAM_FAILED` 或
`CTSTRAFFIC_SINGLE_UDP_STREAM_FAILED` 的硬 `RATE_FAIL`；环境、参数或清理问题仍是 `SETUP_ERROR`。

**Q: 截图没生成，怎么知道失败原因？**
查看 `master_日期时间.log`，里面会写明主控截图 API、辅测 HTTP、JSON/Base64 解码或本地写盘的具体错误。截图失败不改变吞吐测试结论。

**Q: WiFi 网卡没识别出 5G/2.4G？**
WiFi 必须已连上 AP 才能识别频段；没连接的 WiFi 网卡会被忽略。

**Q: 中途断了想续跑？**
用 `--resume`（或 config 里 `resume: true`），24 小时内正式 PASS 过的任务自动跳过。
`MEASURED`、`NOT_EVALUATED`、`RATE_FAIL` 都不会被当成已通过。

```
# Windows 上（装一次 rustup：rustup.rs 下载 rustup-init.exe，默认选项即可）
```

### CI 自动编译

每次推送版本 tag（`v1.2.3`）先执行格式化检查、`cargo test --locked` 和零警告 Clippy；全部通过后才锁定依赖编译 Windows / macOS / Linux 三平台并发布到 GitHub Releases。
Action 路径：`.github/workflows/build.yml`。

### ping 输出解析策略

Windows ping 的统计行（"已发送=4，已接收=4"）可能把 ICMP 错误应答（"无法访问目标主机"、
"TTL 超时"、"一般故障"等）也计为"已接收"；macOS 输出还可能混入 ICMP Redirect。本工具采用
**白名单策略**逐行核查：只有目标回复行同时带 RTT 时间（`时间<1ms` / `time=1.605 ms`）才算真正的 Echo Reply。统计行只提供实际发送数和接收上限，不能单独证明 Ping 成功。

技术要点（对应 PROJECT_PLAN.md）：
- 零 PowerShell：网卡扫描 = ipconfig + GetIfTable2 API + netsh wlan
  （计划里的 wmic 已被 Win11 24H2 移除，故用同等的 GetIfTable2 API，扫描/监控统一）
- 零线程安全坑：无 COM；agent 是固定线程池 + `Arc<Mutex>`，handler panic 不崩服务
- 通信 = HTTP + JSON（tiny_http），带超时/重试/错误码
- iperf3 用文本输出解析（兼容旧版 3.1.x Windows 二进制）；server 保持运行到主控按 request ID 同步 stop，普通地址用 TCP connect 探测就绪并再次确认本次 Child 仍存活
- ctsTraffic 使用 Windows 专用异步作业端点，TCP/UDP 按官方相反角色语义映射并解析 status/summary
- NIC 吞吐监控 = GetIfTable2 InOctets 差值（macOS: netstat -ibn）
